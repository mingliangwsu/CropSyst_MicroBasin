Parameters are introduced to the file. Unlike the version 4 parameter file, parameters are added to the file as desired. Parameters are initially provided with default values the user then introduces parameters that superceed the default values. The parameter file can also 'inherit' parameters from a specified parameter. When the parameter file is initially created this file will be the crop parameter file of the parent directory (this supports the new claudistic idiom)

Parameter entries

Parameters are constructed of a key and value. 
key: value

key

Key are typically specified heirachically corresponding to the path in the crop ontology variable definition directory structure.
For example: 

leaf:
   area:
      index:
         active: GAI value
The definition will be defined by the Technical, Operational, Theoretical, and Conceptional ontological definitions depending on what is available. These definitions will be searched in this order. For example Technical definitions may be omitted if there is no specialization of the Operational, Theoretical, or Conceptional definitions.

Related values are typically grouped using YAML accordingly: 


leaf:
   area:
      index:
         active: GAI value
         inactive: LAI value

In the short form the value is a scalar value. Any properties of the value such as units an quality are assumed to be as defined by the variable ontology definition. Conceptual:
The key can be represented using a shorthand representation, Ie.:
leaf/area/index/active: GAI value
leaf/area/index/inactive: LAI value

Parameters are either intrinsically constant or dynamic. This is defined by the ontology.

Constant parameters

Constant parameters cannot be modified during the course of the growing season. Such parameters include:
  • basic plant characteristics
  • submodel selection
  • thermal time to phenologic stages
  • harvest index
Constant values will be entered in the short form.

Numeric parameter

Numeric entries are entered with YAML map constructs where parameters are key and value pairs. Default numeric entries entered using the short form will have constant continuous value throughout the crop duration.

For numeric values one may want to override the expected properties defined by the technical ontology. In this case instead of using the short form, the value is provided with additional information: 

key: 
   normalization:  value
   quality: observed or calculated or interpolated or other or estimated or forecasted or generated or derived
   units:  units

The key: normalization: value may be provided without additional information.

The normalization entry indicates that the value should be consider the value to be used in normal conditions, but may be preempted a value in superceeding conditions with greater precedence.

units may be omitted when the value conforms to units prescribed by a technical specification of the file/data stream (and/or predefined in the implementation of the program).

It would be preferrable that a program read variable values perform necessary units conversion. Or at least check that specified units for the value conform to expected values and emit an error if this is not the case.

When writing to a data stream it is highly recommended that units be included for the sake of clarity for users that my visually inspect the data where a technical specification might not be provided or readily available. 
key: 
   normalization:  value
is equivelent to key: value

quality

The quality of the value is optional but should be specified. If quality is omitted, it is assumed be observed/measured. CropSyst parameter calibration/preparation utilities may calculate/interpolate or otherwise derive values, in which case the quality will be set accordingly

units

The units are optional, if omit the units are assumed to be the same as the variable ontology definition. If the units are specified and different than the defined units, the value shall be converted as needed accordingly.

Parameter calibration

key: 
   calibration:  value
A parameter may have a value denoted as a calibration. The calibration is a value provided by a calibration routine. For example CropSyst provides utilities to derive calibrated values derived from emperical/observed data.

Parameter adjustment

key: 
   assumption:  value
In addition to normalization and calibration value parameters could be futher adjusted to accomodate cultivars and/or local conditions. For example a crop parameter file may have variables with normalized or calibrated values suitable for a specific location but may need to be further adjusted for use in other locations by other heuristic derivation method(s). I this case the derived value is 'assumed' to be better than any other provided values. Generally assumption entries are to parameter files by an a program that calculates such adjustments. This program should preserve existing normalization and/or calibration entries in the file. Programs using the file will preferentially use the assumed over normal or calibrated values. It is not necessarily the case that adjusted values are of greater quality than calibrated or normal values (indeed normal or calibrated values would usually be considered higher quality since there deriviation is more based on higher quality information, but for the sake of practicality is is nessisary that the model overrides with 'assumed' .

Dynamic parameters

Version 4 had some sets of parameters for values that changed during the season, but a separate parameter had needed to be added for each synchronization event as originally V4 didn't allow for values to change other than at the start of a growth stage (for instance). Version 4 introduced the ability to schedule the adjustment of some parameters during the growing season.

This new YAML structure allows much more flexibility allowing one to specify parameter adjustments thoughout the crop duration. This mechanism has been extended to allow all potentially adjustable parameters to be dynamically re-set using various time series adjustment methods.

Time series values

For most numeric parameters, one can specify thermal-time series data points for interpolated dynamic adjustment of the parameter values set with respect to at phenologic miles stones

The operational ontology definition of the parameter indicates whether the parameter can be so temporarly adjusted. Adjustable parameters are identified as being 'dynamic'. Currently the interpolation time series method types are: step (piecewise constant) and linear. If the method is omitted the interpolation defaults to step
Future versions may implement: piecewise, polnomial, quadradic, Lagrange, Gaussian, spline, cubic_spline conservative

interpolation: !method [ set of time series point values ]
where the time series points value entries absissa are either phenologic milestone, thermal time, or actual or relative point values of the form:
{ sync: days stage: growth-stage offset: day-count ordinate: ordinate-value }

or of the form
{ sync: °C-days stage: growth-stage offset: degree days ordinate: ordinate-value }

or of the form
{ sync: date offset: date in format YYYYDDD ordinate: ordinate-value }
Dates can be either have actual years (> 1000) or relative years where 0 is the sowing year, 1 is the first year after sowing, 2 the second, etc..

or of the form
{ sync: season , offset: season index , ordinate: ordinate-value }
(this form is used for season biomass adjustment in CropSyst)

For example: 

     interpolation: !linear
       - sync: days       
         stage: emergence 
         offset: 4 
         ordinate: 0.01   
       - sync: days            
         stage: maturity 
         offset: 5 
         ordinate: 0.6     
       - sync: days      
         stage: senescence 
         offset: 2 
         ordinate: 0.4    
       - sync: °C-days    
         stage: planting
         offset: 1050 
         ordinate: 0.45      
       - sync: date
         offset: 0000165  #day of year 165
         ordinate: 0.06   
   
     interpolation: !step
       - sync: days
         stage: anthesis 
         offset: 1 
         ordinate: 0.3    
       - sync: °C-days   
         stage: emergence 
         offset: 20 
         ordinate: 0.06
Alternate inline YAML rendering: 
   interpolation: !linear
         - { sync: days,    stage: emergence,  offset: 4,     ordinate: 0.01  }
         - { sync: days,    stage: maturity ,  offset: 5,     ordinate: 0.6   }
         - { sync: days,    stage: senescence, offset: 2,     ordinate: 0.4   }
         - { sync: °C-days, stage: planting ,  offset: 1050,  ordinate: 0.45  }
         - { sync: date,    offset: 0000165, ordinate: 0.06  }
   
   interpolation: !step
         - { sync: days,    stage: anthesis ,  offset: 1,     ordinate: 0.3   }
         - { sync: °C-days, stage: emergence , offset: 20 ,   ordinate: 0.06  }

Here, the variable will be initialized with the scalar value and will be adjusted to the respective values the specified milestones/dates are reached. In the case of step point, The ordinate value will be set at the specified number of days after the starting date of the specified growth stage. The parameter will retain the constant value until the next event occurs.
In the case of linear point, the value will be calculated from the value at the start of the respective growth stage to the thermal time of the start of the growth stage plus the specified degree days.
Note that there could be multiple concurrent curves the curves are sorted/stacked such that the current active curve will be given the priority thermal time of the latest growth stage encountered, Once the point has been encounted, the curve is deleted. Date curves have the lowest priority.
Note there is a space after the colons (and commas) conformant with YAML. The milestones or thermal-time in the time series events may be listed in any order, but it is recommended to list them in natural progressive order.

Both constant phenologic milestone and thermal-time (and actual or relative dates) entries can be included in time series. In this case the constant phenologic milestone values will reset the linear interpolation curve value for the current point and linear interpolation will proceed from the reset value until the next milestone or thermal time or date is achieved.

Observations

The format accomodates the storage of observation/measured records in the crop parameter file. Observations are used by parameter calibration utilities such as the CropSyst calibrator.
Some observation values are stored under the corresponding crop parameter sections with the optional assocation observation.

For example: 


leaf:
   area:
      index:
         active: 
            observation:
               - {1980125: 1.2}
               - {1980132: 1.4}

In this case the time series abscissa values are actual dates. Relative dates can alternatively be specified.

For example: 


leaf:
   area:
      index:
         active: 
            observation: 
               - { 0000125: 1.2} 
               - { 0001132: 1.4}

Date entries are always 7 digits, where the first 4 digits denote the year. Years less than 1000 are assumed to be relative date where 0000 is the year of planting, 0001 the first year after planting, etc. Usually calibration observed data are actual or relative dates, but points relative phenologic growth stage and/or thermal time can also be specified in the time series (there can be mixed milestones, thermal time, actual date and relative dates in the same time series) The number of digits distingishes the type of abscissa, Abscissa values with 7 digits are years, 6 digits or less are assumed to be thermal time and a growth stage name indicates a milestone abscissa.
Conceptual: Observation items can also be phenologic milestone, thermal time, or date point values as with the interpolation items.

Calibration

A parameter with the extended form may have in addition to a normalization value entry observation and/or interpolation (as well as units and quality). Only the normalization entry is required. For CropSyst, observations are generally used only for calibration to derive the calibration value. During the simulation run, the parameter value will be loaded with the normalization value, which should be retained for the duration of the crop. The calibration entry can be used to override this behaviour with the mode options normalization, interpolation, observation indicating which data source should be for the value at the current time step.
normalization
A value to be normally used based on expert knowledge. In CropSyst this would correspond to a value entered in the parameter editor to be used normally through out the duration of the simulation unless superceeded by a value introduced with higher precidence
interpolation
With this mode, the value will be initialized with the normalization value but will be superceded with the interpolated value as the interpolation curve comes into effect.
observation
With this mode or observation values will supercede the normalization value when the respective dates are accounted. If actual dates are listed, the values are only superceded when the exact specified years' dates are encountered by the model. If relative dates are used in an observation, the value will be used whenever the day of year (with the correspond relative year offset) is encountered (during the rotation).
